# Software Overview: archery_parser

## What It Does

`archery_parser` converts archery competition result PDFs — generated by the [Ianseo](https://www.ianseo.net/) competition management system — into structured, arithmetically-verified CSV files. It was built for the Estonian archery competition circuit by Vana-Võidu Vibuklubi.

The tool reads one or more Ianseo qualification protocol PDFs and produces a single CSV where each athlete's scores are expanded into multiple rows: one per end score, one per half-subtotal, and one grand-total row. This format is designed for database import and statistical analysis.

---

## Architecture

The system is a 5-stage sequential pipeline, each stage implemented as an independent module:

```
PDF file(s)
    │
    ▼
 1. reader.py       ── PDF → Word objects with (x, y) bounding boxes
    │
    ▼
 2. detector.py     ── Word lines → CompetitionMeta + RawSection list
    │                   (state machine parses headers, section boundaries, distances)
    ▼
 3. assembler.py    ── RawSection lines → AthleteRecord objects
    │                   (groups multi-line athlete data, extracts scores)
    ▼
 4. transformer.py  ── AthleteRecord → CSVRow list (row expansion + formatting)
    │
    ▼
 5. writer.py       ── Arithmetic verification + UTF-8 CSV output
    │
    ▼
 output.csv
```

**`cli.py`** wires all five stages together and provides the command-line interface via argparse.

### Data Models (`models.py`)

Four dataclasses flow through the pipeline:

| Model | Purpose |
|---|---|
| `CompetitionMeta` | Header metadata: name, organiser, event code, venue, dates |
| `SectionContext` | Per-category metadata: bow type, age class, gender, distances |
| `AthleteRecord` | One athlete's full parsed data: name, club, scores, section ref |
| `CSVRow` | One output row with all 9 CSV columns |

### Lookup Tables (`lookups.py`)

Maps Estonian Ianseo terminology to canonical English strings:
- `BOW_TYPE`: "Sportvibu" → "Recurve", "Plokkvibu" → "Compound", etc.
- `AGE_CLASS` / `GENDER`: Ianseo class codes ("M", "U18W", "50M", "HM") → age/gender strings
- `build_distance_context()`: Derives half-subtotal and grand-total distance labels from per-end distances

## Stage-by-Stage Detail

### Stage 1: Reader (`reader.py`)

Opens a PDF with `pdfplumber`, extracts every word with its bounding box, then groups words into logical lines using y-coordinate proximity (4pt tolerance). Each page is processed independently to avoid cross-page merges. Output: `list[list[Word]]`.

### Stage 2: Detector (`detector.py`)

A 6-state machine scans the line stream:

1. **HEADER** — First 3 lines → `CompetitionMeta` (name, organiser/event code, venue/dates)
2. **BETWEEN** — Waits for "After N Arrows" sentinel
3. **EXPECT_TITLE** — Parses section title (Estonian bow type + age/gender) into a class code
4. **EXPECT_COL_HDR** — Collects distance column headers (tokens like "70m-1", "30m-4")
5. **ATHLETE_DATA** — Accumulates athlete data lines
6. **SKIP_SECTION** — Discards unrecognised sections

Output: `(CompetitionMeta, list[RawSection])` where each `RawSection` holds the category context and raw athlete lines.

### Stage 3: Assembler (`assembler.py`)

Groups printed lines into per-athlete blocks using a two-condition anchor: first token is a positive integer (position) AND second token matches the target-code regex (`\d-\d{3}[A-Z]`). Then parses each block sequentially: position → target code → lastname → firstname → class code → club → scores. Scores are split into end scores, half-subtotals, and grand total using the repeating `[end_a, end_b, subtotal]` pattern.

### Stage 4: Transformer (`transformer.py`)

Converts each `AthleteRecord` into multiple `CSVRow` objects:
- Reformats names to "Firstname Lastname" title case
- Merges club code/name into a single field
- Expands scores: 2-end → 3 rows, 4-end → 7 rows, N-end → N + N/2 + 1 rows
- Assigns distance labels from the section context

### Stage 5: Writer (`writer.py`)

Groups rows by athlete, runs arithmetic verification (half-subtotals = sum of two ends; grand total = sum of halves), logs warnings for any mismatches but never suppresses rows, then writes the CSV.

## Problems Found

### Bugs

1. **`_parse_int` strips legitimate decimal points** (`assembler.py:68`)
   The function `_parse_int` strips both commas and periods before parsing. This means a value like "1.5" would be parsed as "15" rather than failing. While unlikely in this integer-score context, this is a latent bug if any non-integer numeric token ever appears in the input. More importantly, some European locales use periods as thousands separators (e.g., "1.238"), so this may be intentional — but there's no comment explaining the design choice.

2. **`_is_grand_total_row` ignores its own logic for 2-end rounds** (`writer.py:73-88`)
   `_is_half_subtotal_row` returns `True` only when the numeric prefix before "x" is exactly 2. For a 72-arrow round the grand total label is "2x60m" — this is classified as a *half-subtotal*, not a grand total. The verification code in `_verify_athlete_group` falls through to the `if half_rows` branch and attempts to sum half-subtotals that should actually be the grand total. In practice this likely works because a 2-end round has no real half-subtotals, so `half_rows` ends up containing only the grand total and `grand_rows` is empty, meaning the check is simply skipped. The row classification logic is fragile and confusing.

3. **`CSVRow.COLUMNS` is an instance-level mutable list** (`models.py:141-155`)
   The `COLUMNS` field uses `default_factory=lambda: [...]` which creates a new list on every `CSVRow` instance. This should be a class-level constant, not a dataclass field. Every row object carries an unnecessary 9-element list copy. While not a correctness bug, it wastes memory proportional to the number of rows.

4. **Closure over mutable variable in `_finalise` / `_finalise_section`** (`assembler.py:336`, `detector.py:275`)
   Both `assemble_athletes` and `detect_sections` define inner closures (`_finalise`, `_finalise_section`) that close over mutable loop variables. In the assembler, `_finalise` captures `current_group` from the enclosing scope — but the loop rebinds `current_group` as a new list each iteration, so the closure always sees the latest value. This works correctly *only because* `_finalise()` is always called before `current_group` is rebound. It's fragile: if the code is ever refactored to call `_finalise` after the rebind, it would process the wrong group.

### Design Issues

5. **No error handling for corrupt/non-Ianseo PDFs**
   If `pdfplumber` successfully opens a non-Ianseo PDF, the detector will fail with a confusing `ValueError` or silently produce zero sections. There's no early validation that the PDF is actually an Ianseo document (e.g., checking for expected header structure before committing to parsing).

6. **`--output` and `--append` are mutually exclusive but not enforced**
   Both flags can be passed simultaneously. When both are provided, `--append` wins silently (the `if args.append` branch in `cli.py:166` is checked first). This should either be an error or documented.

7. **`sys.exit(1)` inside `_process_pdf` breaks testability** (`cli.py:133,141`)
   The pipeline runner calls `sys.exit(1)` directly on errors, making it impossible to test error paths without catching `SystemExit`. The function should raise exceptions and let `main()` handle exit codes.

8. **Unused `--encoding` argument** (`cli.py:107-111`)
   The `--encoding` flag is defined in the argument parser but never used — `write_csv` always opens with `encoding="utf-8"`. The encoding argument is silently ignored.

9. **`_group_into_lines` uses greedy grouping from the first word's y-midpoint** (`reader.py:84-85`)
   Line grouping compares each word against the *first* word in the current line. For lines with slight y-drift across the page (common in PDFs), a word at the right edge could drift more than 4pt from the left-edge anchor while still being on the same visual line. A running-average or "compare to previous word" strategy would be more robust.

10. **Hardcoded Estonian language assumption**
    The entire detector and lookup system assumes Estonian section titles. Ianseo supports multiple languages. The tool silently skips sections in other languages (via `SKIP_SECTION`), losing data without a clear warning that language was the issue.

11. **No duplicate detection across append operations**
    When using `--append`, the same competition PDF can be processed multiple times, producing duplicate rows. There's no deduplication or warning.

### Code Quality

12. **Inconsistent use of `Counter` import** (`lookups.py:8`)
    `Counter` is imported from `collections` but never used anywhere in the file.

13. **`_is_end_row` has an unused `section_distances` parameter** (`writer.py:44`)
    The function signature accepts `section_distances` but the implementation never references it. This dead parameter is also visible at the call site in `_verify_athlete_group` where it's called as `_is_end_row(row, [])` with an empty list.

14. **Magic strings for state machine states** (`detector.py:265`)
    The state machine uses raw strings like `"HEADER"`, `"BETWEEN"`, etc. An `Enum` would catch typos at import time and make the valid states discoverable.

15. **No type annotations on `build_distance_context` return** (`lookups.py:80`)
    The function returns `dict` but should return `dict[str, list[str] | str]` or, better, a `TypedDict` or small dataclass, since callers access specific keys.

## Improvement Suggestions

### High Priority

1. **Fix the unused `--encoding` flag** — Either pass `args.encoding` through to `write_csv` or remove the CLI argument. This is a user-facing broken promise.

2. **Remove the unused `Counter` import** and the dead `section_distances` parameter — Low-effort cleanup that removes confusion.

3. **Make `CSVRow.COLUMNS` a class constant** — Replace the dataclass field with a plain class variable: `COLUMNS = ["Date", "Athlete", ...]`. Saves memory, communicates intent.

4. **Replace `sys.exit` with exceptions in `_process_pdf`** — Return errors to `main()` and handle exit there. This improves testability and composability.

5. **Enforce mutual exclusivity of `--output` and `--append`** — Use `argparse.add_mutually_exclusive_group()` or add an explicit check.

### Medium Priority

6. **Use an Enum for detector states** — Replace string literals with `class State(enum.Enum)`. Prevents typo bugs and makes the state machine self-documenting.

7. **Add PDF format validation** — Before entering the state machine, check that the PDF has at least the expected header structure. Emit a clear error like "This does not appear to be an Ianseo qualification protocol" rather than cryptic parsing failures.

8. **Add `--strict` mode** — Exit non-zero on arithmetic mismatches instead of just logging warnings. Useful for CI/automated pipelines where data quality must be enforced.

9. **Improve line grouping robustness** — Switch from "compare to first word in line" to "compare to previous word" or a running average for the y-midpoint reference. This would handle gradual y-drift in poorly-generated PDFs.

10. **Add deduplication for append mode** — When appending, check if the (Date, Athlete, Competition, Distance) tuple already exists in the target CSV and warn/skip.

### Low Priority

11. **Support multiple languages** — Make the detector's section-title parsing configurable via a locale/language parameter, or auto-detect the language from the "After N Arrows" sentinel (which may appear in different languages).

12. **Add JSON/SQLite output options** — CSV is limited for structured data. Supporting JSON or direct SQLite insertion would make the tool more useful for data analysis workflows.

13. **Add a `--validate-only` mode** — Like `--dry-run` but also prints a structured report of all parsed data and any verification issues, without writing output.

14. **Add progress reporting for multi-file processing** — When processing many PDFs, log progress per file (currently done, but at INFO level which requires `--verbose`). Consider a progress bar for terminal use.

15. **Document the `_parse_int` thousands-separator handling** — Add a comment explaining why both `.` and `,` are stripped, and under what input conditions this is expected.

## Test Coverage

The project has 223 unit tests covering all five modules. Integration tests are gated on a PDF fixture (`tests/fixtures/Puiatu-CUP-2025.pdf`) that is not committed to the repository. The tests are well-structured, with per-module test files and good coverage of edge cases.

**Testing gaps:**
- No tests for the CLI error paths (`sys.exit` branches)
- No tests for the `--encoding` flag (since it's not implemented)
- No integration tests that can run without a proprietary PDF fixture
- 216-arrow and 288-arrow formats are modelled but not tested against real data
